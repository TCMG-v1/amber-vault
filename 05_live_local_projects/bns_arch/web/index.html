The Four Layers

Read top to bottom. Each layer is its own hardened thing; the seams between them are where the work lives. The money stays boring and compliant. The provenance stays yours.

§ III
The Seams — Where Future-Proofing Lives

"Future-proofing constantly, within reasonable reason." The trick isn't predicting the future — it's clean seams, so any one layer can be swapped without ripping out the others.

Seam A · Store ⇄ Records
The merchant layer emits a plain purchase-confirmed event. Swap processors (Stripe today, something else tomorrow) and nothing downstream cares. The contract is the event, not the vendor.
Seam B · Records ⇄ Ledger
The back office hands the ledger a sealed-work reference + sale fact — never card data. The hash layer is deliberately ignorant of payments. Money side stays compliant; provenance side stays yours.
Seam C · Everything ⇄ Emulator
The same events run against a sandbox. If it works in the emulator, it works in production — because production is just the emulator with a real processor plugged into Seam A.
§ IV
Standing Principles
P · 01
Keep the money boring
Card processing is a hardened, compliant, off-the-shelf thing. Never put your clever ideas inside the payment flow. Ride alongside it.
P · 02
The buy is the seal
A purchase fires a hash and writes a ledger line. Commerce feeds provenance automatically. Selling it is proving it.
P · 03
Build seams, not predictions
Future-proof by making layers swappable, not by guessing what's coming. Reasonable reason: ship the seam, not the prophecy.
P · 04
Prove before live
Nothing touches real money until it's run end-to-end in the emulator. Demonstrable first. It needs to just work.

The store does not merely sell the work; it witnesses it. Every purchase is a hand raised, every hash a name written down. We sort so the work is found; we seal so the work is proven.

◬ So why build it this way

Because the money stays boring and the provenance stays yours. The card processor never touches the hash; the hash never touches the card. They ride side by side, joined only by a clean event. That seam is the whole future-proofing strategy — swap any piece without breaking the others. It needs to just work, and it needs to be demonstrable before a single real dollar moves. This page is that demonstration.

What this page does
  • Runs a real SHA-256 in your browser when you tap — the same primitive a real sale would use.
  • Keeps a tiny in-memory ledger that only ever grows, never rewrites a prior line.

  • Reaches you through the pplxstudio edge — pass-through only, which rewrites the address and forwards. It adds nothing, so every claim above survives the hop.
What this page does NOT do
  • Take any card, payment, or money — there is no checkout here; $0 is ever at risk.
  • Phone home — zero network calls fire from our scripts; instrumented and counted live above.
  • Set a cookie or write to local/session storage — it forgets you the instant you leave.
  • Load any third-party script, tracker, pixel, or analytics beacon.
  • Autoplay audio or motion, or move anything if you've asked for reduced motion.
